# Frontend TaskMaster — Next.js (v2)

## Overview

Создать фронтенд для TaskMaster в отдельном репозитории. Стек: **Next.js 16 + TypeScript**,
**shadcn/ui + Tailwind CSS**, **TanStack Query + Zustand**. Фронтенд общается **только с
Gateway-service** через единый URL — Next.js API routes проксируют запросы на Gateway,
управляют httpOnly-cookie для refresh-токена и пробрасывают access-токен из Zustand.

**Ключевое изменение vs v1:** вместо 6 прямых URL к сервисам — один `NEXT_PUBLIC_GATEWAY_URL`.
Next.js API routes заменяют прямые fetch-вызовы браузера.

## Известные проблемы контрактов

| Проблема | Сервис | Как обходить |
|---|---|---|
| `profileId` в Employee может быть `null` | OrgService | Показывать `userId` если `profileId == null` |
| `GET /api/organizations` возвращает все орги | OrgService | Фильтровать по `GET /api/organizations/user/{userId}/info` |
| `GET /api/tasks` — нет фильтра по org | TaskService | Всегда передавать `?projectId=` |
| Gateway не маршрутизирует `/api/projects/**`, `/api/invitations/**`, `/api/profiles/**`, `/s3/**` | Gateway | **Исправить в Task 0** до старта фронтенда |

## Development Approach

- **Тестирование:** нет
- Каждую задачу завершать полностью до перехода к следующей

- Одна переменная: `NEXT_PUBLIC_GATEWAY_URL`

## Solution Overview

```
Browser
  └─ fetch /api/auth/login      (Next.js API route — управляет cookie)
```

### Task 2: Next.js API routes (auth + catch-all proxy)

**Files:**
- Create: `app/api/auth/login/route.ts`
- Create: `app/api/auth/register/route.ts`
- Create: `app/api/auth/refresh/route.ts`
- Create: `app/api/auth/logout/route.ts`
- Create: `app/api/proxy/[...path]/route.ts`

- [ ] **`app/api/proxy/[...path]/route.ts`** — универсальный catch-all:
  - Читает `Authorization` header из входящего запроса (браузер передаёт access token)
  - Перенаправляет `GET/POST/PUT/DELETE` на `${GATEWAY_URL}/{path}` с теми же body/headers
  - Пробрасывает статус и тело ответа от Gateway без изменений
- [ ] **`app/api/auth/login/route.ts`**:
  - `POST` → вызывает `${GATEWAY_URL}/api/auth/login`
  - Из ответа извлекает `refresh_token` → ставит httpOnly, Secure, SameSite=Strict cookie `refresh_token`
  - Возвращает только `{ access_token, user_id }` клиенту
- [ ] **`app/api/auth/register/route.ts`**:
  - `POST` → вызывает `${GATEWAY_URL}/api/auth/register`, затем автоматически вызывает login
  - Ставит httpOnly cookie, возвращает access token клиенту
- [ ] **`app/api/auth/refresh/route.ts`**:
  - `POST` → читает httpOnly cookie `refresh_token` из запроса
  - Вызывает `${GATEWAY_URL}/api/auth/refresh` с `{ refresh_token }`
  - Обновляет cookie, возвращает новый `access_token`
- [ ] **`app/api/auth/logout/route.ts`**:
  - `POST` → очищает httpOnly cookie (`maxAge: 0`)
  - Возвращает `200 OK`
- [ ] Проверить вручную через curl что login ставит cookie и возвращает access token
